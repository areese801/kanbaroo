# Changelog

All notable changes to Kanberoo are recorded here. This project follows [Semantic Versioning](https://semver.org/) and [Keep a Changelog](https://keepachangelog.com/).

## [Unreleased]

### Added
- TUI scaffold and board view for milestones 11 and 12 (spec section 8) in `kanberoo-tui`. New `kanberoo-tui` console script launches a Textual app that reads `$KANBEROO_CONFIG_DIR/config.toml` (same shape the CLI uses, with `$KANBEROO_API_URL` / `$KANBEROO_TOKEN` overrides) and routes the user from a workspace list into a kanban board for the selected workspace.
- `kanberoo_tui.client.AsyncApiClient`: async `httpx`-based REST client mirroring the CLI's `ApiClient` shape (bearer auth, canonical error envelope translation, ETag/If-Match helper for `POST /stories/{id}/transition`). Accepts `transport=` so tests drive it with `httpx.MockTransport`.
- `kanberoo_tui.ws.EventSubscriber`: async WebSocket subscriber that connects to `ws://.../api/v1/events?token=...`, filters out the `{"type":"ping"}` keepalive frames, and auto-reconnects with exponential backoff. Dependency-injected `connector` callable lets tests script frame sequences without a real socket.
- Workspace list screen (`WorkspaceListScreen`): landing view that fetches every workspace, adds story and epic counts (one extra request per endpoint per workspace, bounded by single-user scale), and renders a `DataTable`. Keybindings: `j`/`k` to move the cursor, `enter`/`l`/`→` to open the board, `r` to refresh, `?` for the help panel, `q` to quit. `workspace.*` WebSocket events trigger a refetch.
- Board screen (`BoardScreen`): five-column kanban layout (Backlog, Todo, In Progress, In Review, Done). Each story card shows the human id, a color-coded priority badge (none/low=blue/medium=yellow/high=red), a one-character actor badge for the last state transition (`h`/`c`/`s`), and optional tag chips. Keybindings: `h`/`l`/`←`/`→` between columns, `j`/`k`/`↓`/`↑` between cards, `m` enters move mode where the next key (`b`/`t`/`p`/`r`/`d`) transitions the focused card via `POST /stories/{id}/transition` with `If-Match`, `esc` cancels move mode, `r` refreshes, `q` returns to the workspace list. `enter` and `/` flash "cage I" placeholders for the story detail and search flows reserved for the next cage. `story.*` WebSocket events trigger a refetch.
- TUI test harness under `packages/kanberoo-tui/tests/`: a `MockApi` fixture backed by `httpx.MockTransport`, a `FakeWsStream` test double that lets tests script WebSocket events deterministically, and a `ws_factory` fixture that wires the fake stream into the app. Nine new tests cover the workspace list (row rendering, `j`/`k` cursor movement, `enter` pushes the board, `workspace.updated` triggers a refetch), the board (stories placed in the correct state columns, `m`+`t` issues the transition with the right `If-Match`, the card moves after the refetch, a `story.created` event drives a refetch that shows the new card in the Backlog column), and the WS subscriber (URL builder for `ws`/`wss`, ping frames filtered out of the event stream).
- Top-level `pyproject.toml` now excludes `packages/kanberoo-tui/tests/` from the workspace-wide mypy run, mirroring the existing carve-out for the CLI tests so both sibling `tests/conftest.py` files coexist without tripping mypy's unique-module-name rule.

- CLI surface for milestones 9 and 10 (spec section 7.2) in `kanberoo-cli`: every non-`init` command is now implemented. New sub-apps: `kb workspace list/create/show`, `kb story list/create/show/edit/move/comment/link/delete`, `kb epic list/create/show/close/reopen`, `kb tag list/create/rename/delete`, `kb token list/create/revoke`, `kb server start/stop`. New top-level commands: `kb audit <KAN-N|UUID>`, `kb export --workspace <KEY> --output <dir>`, `kb backup [--output <dir>]`. Every list/show command supports `--json` for scripting and defaults to Rich-rendered tables otherwise. Destructive commands (`story delete`, `tag delete`, `token revoke`) prompt for confirmation unless `--yes` is passed.
- `kanberoo_cli.config.load_config` helper reads `$KANBEROO_CONFIG_DIR/config.toml` (default `~/.kanberoo/config.toml`) and applies `$KANBEROO_API_URL` / `$KANBEROO_TOKEN` / `$KANBEROO_DATABASE_URL` environment overrides. Missing config exits 1 with a Rich-rendered hint pointing the user at `kb init`.
- `kanberoo_cli.client.ApiClient` is a thin `httpx.Client` wrapper that attaches the bearer token, composes `/api/v1` paths, translates the canonical error envelope into `ApiRequestError`, and provides `patch_with_etag` / `delete_with_etag` / `post_with_etag` helpers that fetch the current ETag via GET and pass it back as `If-Match`. The `transport=` kwarg lets tests swap in an `httpx.MockTransport` without monkey-patching any handler.
- `kanberoo_cli.resolvers` hides human-id resolution: story and epic refs like `KAN-123` go through `GET /stories/by-key/{human_id}` or `GET /epics/by-key/{human_id}`; workspace refs try `GET /workspaces/{key}` first and fall back to walking the list for a key match.
- `kb story edit <KAN-N>` dumps the current description to a temp `.md` file, launches `$EDITOR` (default `vim`) in the `git commit` style, and PATCHes only when the buffer changed. An unchanged buffer short-circuits with "no changes, nothing to do.".
- `kb server start/stop` shells out to `docker compose up -d` / `docker compose down` via `subprocess.run`. `$KANBEROO_COMPOSE_FILE` flows through as `-f` so CI can point at a non-default file. `--wait` on `start` polls `/api/v1/workspaces` until a 2xx or 30 seconds elapse.
- `kb backup` is local-only: parses the SQLite path out of `database_url` and copies it with `shutil.copy2` to `<output>/kanberoo-YYYYMMDD-HHMMSS.db` (default output: `$KANBEROO_CONFIG_DIR/backups`). A non-SQLite `database_url` prints a clear `SQLite only for now` warning and exits 0 without touching the filesystem.
- `kb export` streams `GET /api/v1/workspaces/{id}/export` to `<output>/<key>-<timestamp>.tar.gz`. That endpoint is cage K's responsibility and currently returns 404; the CLI surfaces a friendly "export endpoint not available yet" message on stderr and exits 1 instead of dumping a traceback. The happy-path test uses a canned tarball-shaped byte string via `MockTransport`.
- REST addition: `GET /api/v1/epics/by-key/{human_id}` mirrors the existing story-by-key endpoint so the CLI's `--epic KAN-N` flag can resolve to a UUID without walking a workspace's epic list. Backed by a new `services.epics.get_epic_by_human_id` helper and covered by a roundtrip/404 integration test in `test_epics_api.py`.
- CLI test harness under `packages/kanberoo-cli/tests/`: a `MockApi` fixture builds an `httpx.MockTransport` bound to a scripted request/response table and a `config_dir` fixture writes a temp `config.toml`. 41 new tests cover every sub-command: workspace list (table/empty/JSON)/create/show (key+UUID+404); story list/create/show/edit (no-change short circuit and real PATCH with fake editor script)/move/comment/link/delete; epic list/create/show/close/reopen; tag list/create/rename/delete; token list/create/revoke; audit happy path and not-found; backup (SQLite copy and non-SQLite warning); export (streamed tarball and 404 message); server start/stop with mocked `subprocess.run` and compose-file override.
- WebSocket event stream for milestone 8 (spec section 5): `WS /api/v1/events?token=<plaintext>` publishes live events for every service mutation. Each event carries the full common envelope from spec section 5.3 (`event_id`, `event_type`, `occurred_at`, `actor_type`, `actor_id`, `entity_type`, `entity_id`, `entity_version`, `payload`). Event types match spec section 5.4 (`workspace.created`/`.updated`/`.deleted`, epic/story/comment/tag equivalents, `story.transitioned`, `story.commented`, `story.tag_added`/`.tag_removed`, `story.linked`/`.unlinked`). Tokens are not events (consistent with the no-audit-on-tokens invariant).
- `kanberoo-core.events` in-process pub/sub bus with async-friendly subscribe/publish and bounded per-subscriber queues (drop-on-full, client refetches via REST). `services.events.publish_event` is the sync helper used by every service.
- Events are emitted only after successful transaction commit (via SQLAlchemy `after_commit` listener), so rollbacks never leak phantom events to subscribers. Rollback-no-event is covered by an explicit test.
- 30-second keepalive ping (`{"type":"ping","ts":"<iso>"}`) on the WebSocket so clients can distinguish idle-but-connected from dead connections. Ping interval is module-configurable for tests.
- Integration tests cover happy-path event delivery, auth rejection (close code 1008 on missing/invalid token), and keepalive pings. Service-level tests verify every mutation that emits audit also publishes the matching event, and that rollbacks emit neither.
- REST endpoints for milestone 6 (spec section 4.2): `GET/POST /api/v1/stories/{id}/comments`, `GET/PATCH/DELETE /api/v1/comments/{id}` (ETag + If-Match); `GET/POST /api/v1/workspaces/{id}/tags`, `PATCH/DELETE /api/v1/tags/{id}` (no If-Match because tags do not carry a `version` column per spec section 3.3); `GET /api/v1/stories/{id}/linkages` (union of incoming and outgoing), `POST /api/v1/linkages`, `DELETE /api/v1/linkages/{id}` (also no If-Match: linkages have no `version` column). Story list gains a `?tag=<name>` filter joining `story_tags`/`tags` scoped to the workspace; `POST /api/v1/stories/{id}/tags` attaches tags (idempotent); `DELETE /api/v1/stories/{id}/tags/{tag_id}` detaches.
- `kanberoo-core` service layer for comments, tags, and linkages. `services.comments` enforces one-level threading (replies cannot have replies) and actor-stamped authorship; soft-delete does not cascade to replies. `services.tags` enforces `(workspace_id, name)` uniqueness, supports rename/recolor, and on soft-delete detaches every `story_tags` association in the same transaction. `services.linkages` implements automatic mirroring for both `blocks` ↔ `is_blocked_by` and `duplicates` ↔ `is_duplicated_by` pairs on create and delete (`relates_to` has no paired opposite and is stored unidirectionally), rejects self-linkage and duplicates, and allows cross-workspace linkages per spec section 10 Q2.
- Audit coverage invariant extended to comments, tags, and linkages: every comment mutation writes exactly one attributed `audit_events` row; every tag mutation (create, update, soft_delete) writes exactly one row, with the story-tag cascade on delete treated as an implementation detail (no per-story audit rows). Adding or removing a tag on a story emits one `tag_added`/`tag_removed` row against the story. Linkage create/delete emits exactly one row against the forward end; the mirror cascade is silent.
- Integration tests for comments, tags, and linkages; extended story integration tests cover the `tag` filter, `POST /stories/{id}/tags`, and `DELETE /stories/{id}/tags/{tag_id}`.
- REST endpoints for milestone 5 (spec section 4.2): `GET/POST /api/v1/workspaces/{id}/epics`, `GET/PATCH/DELETE /api/v1/epics/{id}`, `POST /api/v1/epics/{id}/close`, `POST /api/v1/epics/{id}/reopen`; `GET/POST /api/v1/workspaces/{id}/stories` with `state`/`priority`/`epic_id` filters, `GET /api/v1/stories/{id}`, `GET /api/v1/stories/by-key/{human_id}`, `PATCH/DELETE /api/v1/stories/{id}`, and `POST /api/v1/stories/{id}/transition`. Every mutating endpoint requires `If-Match` and returns `ETag`; create responses also return `Location`.
- `kanberoo-core` service layer for epics and stories: `services.epics` covers epic CRUD plus idempotent `close_epic`/`reopen_epic` wrappers; `services.stories` covers story CRUD, by-human-id lookup, filterable list, and the state-machine `transition_story` function. A module-level `_ALLOWED_TRANSITIONS` table encodes the spec section 4.3 state machine. `InvalidStateTransitionError` is a new `ValidationError` subclass carrying `from_state`/`to_state` details so clients can react programmatically.
- `StoryTransitionRequest` Pydantic schema (`to_state`, `reason?`) for the transition endpoint.
- Cross-workspace epic reassignment is rejected (per spec section 10 Q3): creating or patching a story with an `epic_id` in a different workspace returns `400 validation_error`.
- Audit coverage invariant extended: every epic and story mutation writes exactly one attributed `audit_events` row from the service layer. The transition endpoint writes `action="state_changed"` with the caller's `reason` surfaced under `transition_reason` in the audit diff's `after` object.
- Integration tests for `/epics` and `/stories` mirroring the workspace suite: ETag/If-Match/412 semantics, canonical error envelope, cursor pagination across 150 stories, filter combinations, transition happy path and invalid-target rejection, no-op transition rejection, and direct audit-row assertions.
- `kanberoo-core` service layer (`kanberoo_core.services`): `services.audit.emit_audit` writes an immutable `audit_events` row inside the caller's transaction with a full `{"before": ..., "after": ...}` diff; `services.workspaces` covers workspace CRUD (create, list with opaque cursor pagination, get, patch, soft delete); `services.tokens` wraps the auth helpers for use by REST handlers. Service-level domain exceptions (`NotFoundError`, `VersionConflictError`, `ValidationError`) decouple services from transport.
- `ApiTokenCreate` and `ApiTokenCreatedRead` Pydantic schemas for the token surface. `ApiTokenCreatedRead` carries the plaintext in the create response only; subsequent reads return the masked `ApiTokenRead`.
- `kanberoo-api` FastAPI application: `create_app` factory, `kanberoo-api` console script entry point (uvicorn), request-scoped session dependency that owns transaction boundaries, bearer-token auth dependency that resolves to an `Actor`, and `If-Match` / `ETag` helpers.
- REST endpoints for milestone 4 (spec section 4.2): `GET/POST /api/v1/workspaces`, `GET/PATCH/DELETE /api/v1/workspaces/{id}`, `GET/POST /api/v1/tokens`, `DELETE /api/v1/tokens/{id}`. Every mutating endpoint requires `If-Match` and returns `ETag`; create responses also return `Location`.
- Canonical error envelope (`{"error": {"code", "message", "details"}}`) wired across 400, 401, 404, 412, and 500 via FastAPI exception handlers. Domain errors raised by services translate directly into this shape.
- Audit coverage invariant: every workspace mutation (create, update, soft delete) writes exactly one attributed `audit_events` row from the service layer. Token operations are deliberately excluded (per spec section 3.3) and covered by a negative test.
- Integration test suite for the REST surface (`packages/kanberoo-api/tests/`): workspace CRUD with ETag/If-Match/412 semantics, cursor pagination across 150 rows, soft-delete visibility, token plaintext-once contract, revoked-token auth rejection, the canonical error shape on every status class, and the no-audit-on-tokens invariant.
- Ruff `extend-immutable-calls` for `fastapi.Depends/Query/Path/Header/Body/Form/Cookie` and `typer.Option/Argument`, so B008 does not flag standard FastAPI/Typer dependency-injection idioms.
- Initial repo scaffold: uv workspace with five packages (`kanberoo-core`, `kanberoo-api`, `kanberoo-tui`, `kanberoo-cli`, `kanberoo-mcp`) and a top-level `kanberoo` meta-package with an `[all]` extra.
- Dockerfile and docker-compose.yml for local development.
- Ruff, mypy, and pytest configuration at the workspace root.
- Design docs: `docs/spec.md` and `docs/future-skill-draft.md`.
- Branch protection on `main`.
- `kanberoo-core` data layer: SQLAlchemy 2.x ORM models for every entity in spec section 3.3 (workspaces, workspace_repos, epics, stories, linkages, comments, tags, story_tags, audit_events, api_tokens) with UUID v7 primary keys, ISO 8601 UTC timestamps, soft delete, and SQLAlchemy `version_id_col` optimistic concurrency.
- Pydantic v2 Create/Update/Read schemas for every mutable entity, plus Read-only schemas for `audit_events` and `api_tokens`.
- Atomic `generate_human_id` helper that bumps `workspaces.next_issue_num` under `SELECT ... FOR UPDATE` and returns the next `{KEY}-{N}` identifier (shared across stories and epics).
- Initial Alembic migration (`0001_initial`) producing the full spec section 3.3 schema, with partial indexes on soft-delete columns and CHECK constraints on every fixed-domain enum column.
- `kanberoo-core` auth layer: frozen `Actor` dataclass plus `hash_token`, `generate_token_plaintext`, `create_token`, `validate_token`, and `revoke_token` helpers, with SHA-256 hashing and a `kbr_` plaintext prefix. Plaintext tokens are never persisted; only the hash is stored.
- `kanberoo-core` exposes `migrations.upgrade_to_head` so the CLI can run Alembic programmatically without depending on `alembic.ini` location; the `alembic/` directory is also force-included into the wheel.
- `kanberoo-cli` Typer scaffold with `kanberoo` and `kb` entry points, plus a `kb init` command that creates `$KANBEROO_CONFIG_DIR` (default `~/.kanberoo`), applies migrations to a fresh SQLite database, issues the first human API token, writes `config.toml`, and prints the plaintext exactly once.

### Changed
- `pyproject.toml` excludes `packages/kanberoo-cli/tests/` from the workspace-wide mypy run because its `conftest.py` would otherwise collide with the existing `packages/kanberoo-api/tests/conftest.py` under mypy's unique-module-name rule. Both conftests are still exercised by pytest; only the mypy call ignores the cli tests tree. Existing core-tests type coverage is unaffected.

### Fixed
- Model `Enum` columns now store the lowercase `.value` (e.g. `human`) via a shared `enum_values` helper, matching the spec and the Alembic migration CHECK constraints. Before this fix, SQLAlchemy defaulted to the enum `name` (e.g. `HUMAN`), which diverged from the migration-created schema and would reject inserts on a freshly-migrated database.
